# Async Request/Result Pattern

Every long-running API mutation follows the same async pattern.
The handler returns 202 immediately; a `request.result` event on
`/v0/events/stream` signals completion or failure.

## The request_id

A `request_id` is **just a unique correlation number**. It is not a
session ID, not a bead ID, not a resource identifier. It exists
solely to let clients match a 202 response to the corresponding
`request.result` event on the SSE stream. Generated by the handler
with `newRequestID()` (crypto/rand hex with `req-` prefix).

## The pattern

```
Client                        Handler                    Internal OM
  |                             |                            |
  |--- POST mutation ---------->|                            |
  |                             | validate (sync, fast)      |
  |                             | generate request_id        |
  |<-- 202 { request_id } -----|                            |
  |                             | go func() {               |
  |                             |   result, err := om(...)  ->| (unchanged sync OM)
  |                             |   emit request.result      |
  |                             | }()                        |
  |                             |                            |
  |--- GET /v0/events/stream -->|                            |
  |<-- request.result { request_id, status, resource_id }    |
```

## How it works

Take any synchronous POST handler. The old implementation looked
like this:

```go
func handler(ctx, input) (*Output, error) {
    // validate
    // call OM synchronously (may block 30-90s)
    result, err := om.DoWork(ctx, ...)
    if err != nil { return nil, mapError(err) }
    // build response with resource data
    return &Output{Body: result}, nil
}
```

The async version wraps the OM call in a goroutine:

```go
func handler(ctx, input) (*Output, error) {
    // validate (same as before — sync, fast)
    reqID := newRequestID()
    go func() {
        // EXACT SAME OM call, unchanged
        result, err := om.DoWork(context.Background(), ...)
        if err != nil {
            emitRequestResult(reqID, op, "failed", "", err)
        } else {
            emitRequestResult(reqID, op, "succeeded", result.ID, "")
        }
    }()
    return &Output{Body: {RequestID: reqID}}, nil
}
```

The OM call is identical. The only change is WHERE it runs (in a
goroutine instead of inline) and WHAT the handler returns (just
the request_id instead of domain data).

## The 202 response

The 202 response body contains ONLY the `request_id`:

```json
{ "request_id": "req-a1b2c3d4e5f6a1b2c3d4e5f6" }
```

No resource IDs. No session data. No domain fields. The resource
does not exist yet. Returning an ID for it invites the client to
use it before it's ready, causing errors like "session not found"
or "state creating does not accept command suspend."

The client gets the resource ID from the `request.result` event
AFTER the operation succeeds.

## The `request.result` event

One event type for all async operations. Clients filter by
`request_id` to correlate.

Success:
```json
{
  "type": "request.result",
  "payload": {
    "request_id": "req-...",
    "operation": "session.create",
    "status": "succeeded",
    "resource_id": "gc-42"
  }
}
```

Failure:
```json
{
  "type": "request.result",
  "payload": {
    "request_id": "req-...",
    "operation": "session.create",
    "status": "failed",
    "error_code": "create_failed",
    "error_message": "provider startup failed"
  }
}
```

### Payload fields

| Field | Type | Description |
|-------|------|-------------|
| `request_id` | string | Correlation number from the 202 response |
| `operation` | string | `city.create`, `city.unregister`, `session.create`, `session.message`, `session.submit` |
| `status` | string | `succeeded` or `failed` |
| `resource_id` | string | Created/affected resource ID — only on success |
| `error_code` | string | Machine-readable code — only on failure |
| `error_message` | string | Human-readable description — only on failure |

## Operations using this pattern

| Endpoint | Operation |
|----------|-----------|
| `POST /v0/city` | `city.create` |
| `POST /v0/city/{name}/unregister` | `city.unregister` |
| `POST /v0/city/{city}/sessions` (all kinds) | `session.create` |
| `POST /v0/city/{city}/session/{id}/messages` | `session.message` |
| `POST /v0/city/{city}/session/{id}/submit` | `session.submit` |

## Client contract

1. Subscribe to `/v0/events/stream`.
2. Send the mutation POST.
3. Parse the 202 response; extract `request_id`.
4. Wait for a `request.result` event where `payload.request_id`
   matches.
5. Check `payload.status`: `succeeded` or `failed`.
6. On success, use `payload.resource_id` to access the resource.
7. Do NOT use the resource before `request.result` arrives.

## Implementation rules

1. **The goroutine runs the EXACT SAME OM code the old sync
   handler ran.** No changes to the OM. No new queuing mechanisms.
   No reconciler modifications. Take the old synchronous handler
   body, put it in `go func() { ... }()`, emit the result event.

2. **The request_id is just a correlation number.** Generated by
   the handler, returned in the 202, passed to `emitRequestResult`.
   The OM has no knowledge of it.

3. **Use `context.Background()` in the goroutine.** The HTTP
   request context is cancelled when the 202 is sent.

4. **The 202 response contains ONLY `request_id`.** Nothing else.

5. **Emit `request.result` on both success and failure.** The
   goroutine must never silently complete without emitting.

6. **City lifecycle events go to the supervisor event source.**
   The city doesn't exist during create and goes away during
   unregister.

7. **Session events go to the city event provider.**
   The city exists when session operations run.
